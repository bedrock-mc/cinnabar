# Vanilla Texture Vertical Slice Design

**Date:** 2026-07-10

**Branch:** `phase2-textures`

**Status:** Approved

## Outcome

Replace Phase 0 runtime-ID debug colours with Minecraft Bedrock's matching vanilla block
textures while preserving the packed-quad renderer and its memory/performance architecture.
This is the first vertical slice of Phase 2, not a shortcut around it: the asset interfaces are
pack-stack-aware so later server resource packs, non-cube models, transparency, animation,
lighting, and biome tinting extend the same pipeline rather than replacing it.

The first visible milestone textures ordinary axis-aligned full-cube blocks with correct
per-face texture selection, orientation, greedy-quad UV repetition, and per-layer mipmaps.
Unsupported block geometry remains explicitly diagnostic during development. The debug fallback
must be eliminated before the Phase 2 visual-parity gate.

## Source and pinning

- Use Mojang's `bedrock-samples` **full** release asset
  `v1.26.30.32-preview`, source commit `020f1cf`, as the local vanilla resource-pack input
  matching the pinned 1.26.30 protocol family.
- Pin archive `bedrock-samples-v1.26.30.32-preview-full.zip` by SHA-256
  `12d5cddc03acd507e9e0bd412f2e94d34d0a1a855758af7a9eef61b03630ad7c`.
- Record release tag, source commit, archive SHA-256, and the accepted Minecraft EULA in a
  checked-in source manifest.
- **No raw or derived Mojang asset payload is committed to Git.** The full archive, extracted
  PNG/JSON sources, generated texture-array pixels, and compiled runtime blob remain under ignored
  `.local/assets` or build-output directories. Git contains only importer/compiler code,
  provenance manifests and hashes, generated Dragonfly registry metadata, and synthetic test
  assets created for this project.
- The deterministic compiler output is a versioned runtime asset blob plus a manifest containing
  its own SHA-256. Development and packaging generate it from the user's verified local cache;
  distribution of packaged Mojang content remains subject to the Minecraft EULA.
- Runtime/network block truth comes from a deterministic Dragonfly export generated by
  `tools/registrygen`; numeric runtime IDs are never hand-maintained.

## Architecture

### 1. Pack-aware asset crate

Add `crates/assets` with three narrow layers:

1. **Source readers** parse `blocks.json`, `textures/terrain_texture.json`, block model JSON,
   flipbook metadata, and PNG metadata from an ordered pack stack.
2. **Compiler** resolves block state + face to a canonical texture key, normalizes source images
   into equal-sized RGBA array layers, generates mipmaps independently per layer, classifies
   material mode, and emits deterministic binary tables.
3. **Runtime loader** validates magic/version/hash/bounds and exposes immutable, compact lookup
   tables plus the texture-array payload. It performs no JSON parsing during play.

The base vanilla pack is stack entry zero. Later server packs override keys through the same
compiler interface in Phase 6.

### 2. Registry export and lookup

`tools/registrygen` consumes the pinned Dragonfly registry and emits a sorted binary mapping:

`network value -> canonical block state -> six face descriptors`.

Each face descriptor contains a deterministic compact material ID. Its material-table entry holds
the texture-array layer, UV rotation/reflection, material class, and future tint/animation flags.
Both sequential runtime IDs and high-bit network hashes map through the exported registry
metadata. The StartGame `block_network_ids_are_hashes` flag selects exactly one lookup mode for
the session; the runtime never guesses by numeric range or falls through from one mode to the
other. Unknown values resolve to one conspicuous diagnostic material and emit a rate-limited
counter; they never index GPU memory unchecked.

### 3. Packed mesh contract

Keep exactly one eight-byte record (two `u32` words) per greedy quad. The geometry word remains
unchanged. The second word becomes the compact material ID resolved from network value + face
during meshing; the complete network value remains in the paletted world store and immutable
registry map, rather than being duplicated in GPU geometry. A shared material storage buffer maps
that ID to texture layer and flags, so new visual attributes do not widen every quad or create
CPU vertices.

Meshing resolves material equality before merging, so coplanar faces merge only when texture,
UV transform, transparency class, and baked visual attributes agree. UVs are reconstructed from
quad extents in WGSL and repeat at block scale; no stitched atlas is introduced.

### 4. GPU texture path

Extend the existing single chunk bind group with:

- one `texture_2d_array` containing equal-size RGBA layers and independent mip chains;
- one sampler configured for repeat addressing and mip filtering;
- one compact read-only material table.

The custom chunk render phase, global quad/origin/indirect arenas, and MDI/direct fallback remain
unchanged. Uploads are staged through the existing bounded nearest-first budget. Texture data is
uploaded once per compiled pack revision, not once per chunk.

### 5. Staged material support

The initial visible slice renders full-cube opaque blocks. The same descriptor schema reserves
material classes for cutout and blend, plus flipbook/tint flags. Follow-up Phase 2 tasks add:

1. cutout and blended render subphases;
2. non-cube/static block models;
3. flipbook layer selection;
4. biome tinting and client-side block/sky lighting;
5. sky, fog, and clouds.

No temporary API may require rebuilding the chunk renderer around per-subchunk Bevy
`Mesh`/`StandardMaterial` objects.

## Data flow

```text
pinned full bedrock-samples + Dragonfly registry
        -> registrygen / asset compiler
        -> versioned registry map + texture-array blob
        -> validated immutable runtime assets
        -> worker mesher resolves face descriptors
        -> PackedQuad/material lookup
        -> bounded global GPU upload
        -> shared WGSL pipeline samples texture array
```

World streaming stays paletted. Asset resolution occurs while meshing palette entries/faces and
never expands a subchunk into a flat block array.

## Failure handling

- Missing source files, conflicting keys, non-square/unsupported images, invalid model
  references, duplicate registry keys, and nondeterministic output order are compiler errors with
  source paths.
- Runtime blobs use checked lengths, versioning, hashes, and allocation ceilings; corrupt or
  incompatible blobs fail startup with a useful regeneration command.
- GPU array-layer/size/format limits are checked before allocation. Unsupported hardware reports
  the exact required and available limits; it does not silently downsample or fall back to an
  atlas.
- Missing individual mappings use the diagnostic layer and counters so a live world remains
  inspectable while parity gaps are fixed.

## Testing

All behavior follows RED -> GREEN -> REFACTOR.

- Parser/compiler golden tests cover per-face mappings, UV transforms, pack override order,
  deterministic bytes/hashes, missing references, image normalization, mip isolation, flipbook
  metadata, and the registry's sequential/hash modes.
- Runtime tests reject malformed headers, oversized counts, bad hashes, and out-of-range layers.
- Mesher tests prove greedy merges split on differing material descriptors and preserve block-size
  UV repetition for large quads.
- Shader/plugin tests validate WGSL, bind-group layout, texture-layer sampling, material lookup,
  adapter-limit rejection, and unchanged MDI/direct selection.
- A small checked-in synthetic pack supplies test images; Mojang source and compiled texture
  payloads remain in ignored local/build caches.
- Integration verification runs the current live BDS world, records exact asset hashes and
  metrics, and compares representative blocks against the matching vanilla reference at multiple
  distances and angles using Computer Use.

## Performance and acceptance

- Preserve the Task 7 bounded decode/mesh/upload queues and packed/paletted runtime storage.
- Combined client + core RSS remains at or below 650 MB steady state at radius 16.
- Steady-state CPU remains at or below 15% total on the reference class; join/teleport work
  settles within approximately two seconds.
- Texture memory is reported separately, including all mip levels and material tables.
- The slice is integration-ready when common full-cube blocks show correct vanilla top/side/bottom
  textures, orientations, repeating greedy UVs, and clean mip transitions with zero debug colours
  for supported blocks.
- Phase 2 is not complete until all diagnostic fallbacks are removed or adjudicated and the full
  visual-parity checklist in `plan.md` passes.

## Parallel integration boundary

This work is isolated on `phase2-textures`. It may proceed alongside Task 8 because it does not
modify protocol conformance fixtures, Go relay/core lifecycle, or acceptance orchestration.
Before integration, rebase onto the accepted Phase 0/Phase 1 head, regenerate the review package,
run the full Rust/Go suites, and perform a dedicated renderer review. Any overlapping changes to
`app/src/main.rs`, `crates/render/src/plugin.rs`, or `chunk.wgsl` are reconciled only after
the Task 8 branch is clean.
