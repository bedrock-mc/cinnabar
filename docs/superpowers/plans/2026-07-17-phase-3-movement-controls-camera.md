# Phase 3 Movement, Controls, and Camera Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Complete Phase 3 with authoritative collision metadata, bedsim-compatible local movement and correction replay, one semantic keyboard/mouse/controller/touch input path, native held-jump behavior, all three camera perspectives with fail-closed collision, and server-authoritative outbound movement that passes deterministic and live gates without ever transmitting free-camera motion.

**Architecture:** Keep simulation and collision policy deterministic and Bevy-free in `crates/sim`; add a BREG-bound `PREG1001` physics carrier in `crates/assets` generated by the pinned `tools/registrygen` sources; add world-revision identities in `crates/world`; and introduce a small renderer-independent `crates/input` crate for the frozen semantic action and typed-binding contract. Phase 3 owns device-adapter and camera feature modules, but the integration lane exclusively assembles app schedules, shared module roots, manifests, architecture policy, and roadmap changes. Phase 3 publishes an immutable local pose/visibility carrier; Phase 4 alone combines it with rig, skin, equipment, and action state and owns all actor/viewmodel drawing. Every prediction tick records the exact collision-registry and chunk revisions it observed, and correction replay is transactional: replay only against matching identities, otherwise snap and re-anchor. Normal gameplay authorizes `Physics` only after candidate wiring, deterministic review, candidate live evidence, and a separate final-enable gate; `--auto-fly` remains `FreeCamera` and network-silent.

**Tech Stack:** Rust 1.93.1 edition 2024, Bevy 0.18.1, existing `world`/`assets`/`protocol`/`render` crates, a new no-Bevy `input` crate, Go `tools/registrygen`, PowerShell acceptance scripts, checked-in JSONL conformance fixtures, Cargo tests/Clippy/fmt, Go tests/vet, and the architecture-policy tool.

## Global Constraints

- Implementation starts from clean commit `b2940086fa2bc8d7089ae065da575086557cbd67`, whose only delta from the approved functional base `d8e469979a0ec6c4798bb2ffc1dc45d3a9891eeb` is the approved completion design.
- This is strict completion work. Every movement stratum, device class, perspective, correction path, and evidence gate named below is required; do not replace an item with a deferred note.
- Keep the protocol pinned at 1001. A protocol bump is a separate program decision and must not occur in this phase.
- Never commit Mojang, PMMP, Dragonfly, Prismarine, Valentine, server, or native-client source assets. Only deterministic generated metadata, hashes, provenance, synthetic fixtures, and authored evidence may enter git.
- All metadata, histories, action queues, touch contacts, controller slots, collision queries, camera sweeps, network queues, and evidence records are explicitly bounded. Reject non-finite values before mutation.
- Missing, stale, contradictory, out-of-range, unloaded, or unsupported collision data fails closed. Simulation must remain transactional on failure.
- `FreeCamera` can never enqueue, retry, or send `PlayerAuthInput`. StartGame, MovePlayer, corrections, focus changes, and camera perspective changes cannot elevate it.
- Interaction targeting always uses the interpolated local player's vanilla eye and look vector. A displaced third-person camera is presentation only.
- One semantic press may trigger one perspective cycle. A held jump remains held at fixed-tick sampling and may cause a later grounded re-jump; wire press/release flags remain physical edges.
- UI focus, window focus loss, controller disconnect, session replacement, dimension replacement, and movement-authority replacement publish semantic releases before the next simulation tick.
- Existing architecture limits remain binding: production Rust files at most 1,000 lines, module roots at most 300, tests at most 1,200, PowerShell at most 800. Split modules before crossing a limit.
- Use at most three implementation lanes plus one integration lane. A lane owns its listed files until its commit is merged; shared hotspots are serialized as specified under “Cross-lane freezes and dependencies.”
- Each task follows RED → GREEN → focused verification → commit. Do not combine task commits, and do not claim a gate from a test that was not run in that commit.
